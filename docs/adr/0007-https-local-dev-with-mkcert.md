# ADR 0007: HTTPS-only local dev for Realm via mkcert and custom hostname

- **Status:** Accepted
- **Date:** 2026-05-30

## Context

`apps/realm` communicates with `apps/gatekeeper` (the auth server) at runtime. Once Realm runs on HTTPS, the browser's mixed-content policy blocks any request to an HTTP Gatekeeper origin — auth would silently break. Additionally, cookie security attributes (`Secure`, `SameSite`) and certain browser APIs (e.g. `navigator.credentials`) only activate over HTTPS, making a local HTTP dev server an inaccurate representation of the production environment.

Two approaches were considered:

- **Plain `localhost` over HTTP** — zero setup friction, but no HTTPS parity. Mixed-content issues only surface when Gatekeeper is also running, and browser security features behave differently.
- **Custom hostname over HTTPS via mkcert** — requires a one-time setup (mkcert install, `hosts` file entry, cert generation) but gives an origin that matches production conventions and eliminates mixed-content problems from the start.

## Decision

Serve `apps/realm` exclusively over HTTPS at `https://localhost.www.dndmapp.dev:4000` using a self-signed certificate generated by mkcert. No HTTP fallback is provided in the Angular dev server configuration. `apps/gatekeeper` will follow the same pattern at `https://localhost.auth.dndmapp.dev`.

The certificate covers both SANs and lives at `certs/local.pem` / `certs/local-key.pem` (repo root, gitignored). A root `package.json` script (`certs:generate`) wraps the mkcert invocation.

## Developer setup prerequisites (manual, one-time)

1. Install mkcert: `winget install mkcert` (or via Chocolatey/Scoop)
2. Trust the local CA (requires admin): `mkcert -install`
3. Add to `C:\Windows\System32\drivers\etc\hosts` (requires admin):

   ```text
   127.0.0.1 localhost.www.dndmapp.dev localhost.auth.dndmapp.dev
   ```

4. Generate the certificate: `pnpm certs:generate`

## Consequences

- All developers must complete the one-time setup before running `ng serve realm`. There is no HTTP fallback.
- Browser trust is established by the mkcert local CA, so no security warnings appear.
- Cookie and browser security API behavior in dev matches production.
- Each developer's cert is machine-local and is not committed to the repo.
